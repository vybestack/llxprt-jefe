## LLxprt Code Added Memories
- CRITICAL: NEVER delete, remove, or modify the .llxprt/ directory or any of its contents (LLXPRT.md, settings.json, skills/, commands/). This directory contains project memories, settings, and skills that are version-controlled. Do not run rm, git clean, git checkout, or any other command that would remove these files. If git status shows .llxprt files as modified, leave them alone — they are supposed to be there. "Clean workspace" means only: checkout main and pull from origin. It does NOT mean deleting files.
- This is a Rust project (vybestack/llxprt-jefe). Before checking in code changes from the main project directory, run the full verification suite: `cargo fmt --all --check`, `cargo clippy --workspace --all-targets --all-features -- -D warnings`, `cargo build --workspace --all-features --locked`, and `cargo test --workspace --all-features --locked` — or simply `make ci-check` which runs fmt check, the clippy gates, coverage (--fail-under-lines 30), build, and test. Fix any errors and ensure code is formatted before pushing/creating PRs. Always use gh for PRs/issues/comments (never webfetch). For fast iteration use `make quick-check` (cargo fmt + cargo check -q + cargo test -q).
- When addressing GitHub issue #NUM, follow the canonical bounded workflow in `dev-docs/workflow/ISSUE-DELIVERY.md`. Start from current main, create a dedicated issue branch, fetch the issue and comments with gh, and shape a decision-complete acceptance matrix with explicit non-goals, bounded vertical slices, expected paths, and a scope ledger before implementation. Use behavioral RED -> GREEN -> REFACTOR and preserve the ownership boundaries in `dev-docs/project-standards.md`. Stop for user approval before adding an unplanned subsystem or public abstraction, changing workflow/agent-memory/quality tooling or dependencies, moving unrelated tests, implementing behavior absent from the acceptance matrix, or crossing the hard scope budget. Target no more than 25 files or 1,500 net changed lines per PR; perform a mandatory scope review above either target and stop without explicit approval above 40 files or 2,500 net lines. Classify review findings as Blocker-Fix, In-scope-Fix, Reject, or Defer; valid out-of-scope improvements become follow-ups rather than automatic expansion. Commit green slices, run required exact-head verification, open a detailed issue-linked PR, confirm correct ancestry and no conflicts, and watch required CI to completion. Finish when accepted behavior is proven, exact-head gates pass, required reviews are complete and triaged, and the scope ledger is clean; do not continue optional hardening afterward.
- Rust quality baseline (from dev-docs/RULES.md and project-standards.md): TDD is mandatory; prefer explicit domain types over primitive obsession; use Result/Option and typed errors; no unwrap/expect in production paths; unsafe is forbidden unless explicitly approved and isolated; keep side effects at boundary modules and state-transition/reducer logic deterministic.
- When starting work on a new issue in this project, create a dedicated issue branch such as issue72 before making changes.
- For any new feature exposed through the UI, use test-first coverage that includes both UI/unit behavior tests and TUI harness scenario scripts. Create/update the TUI scenario first and prove the relevant test/scenario fails before implementing the feature.
- if you create a pr or push to an existing one always check if there are conflicts on the pr. if so resolve those before telling the user the pr is read.
- Open Code Review is capped at four runs per issue/PR effort: no more than two OCR reviews before opening the PR and no more than two OCR reviews after the PR is opened.
