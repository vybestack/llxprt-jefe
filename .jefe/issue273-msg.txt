Unify Issue and PR comment pagination onto PaginatedList (#273)

Eliminates the duplicated ad-hoc comment-pagination machinery on the Issue
and PR detail screens. Both IssueDetail and PullRequestDetail now store a
shared PaginatedList<IssueComment, CommentDetailIdentity> instead of the
parallel (comments vec + has_more_comments + comments_cursor) triple, and
the per-screen comments_page_pending / next_comments_page_request_id state
is gone (the container owns pending + request-id correlation).

Changes:
- New CommentDetailIdentity { scope_repo_id, number } in domain/pagination
  (one struct for both screens; issue_number/pr_number are both u64).
- IssueDetail.comments and PullRequestDetail.comments are now
  PaginatedList<IssueComment, CommentDetailIdentity>; has_more_comments and
  comments_cursor fields removed (derived from next_page).
- Deleted IssueCommentsPagePending and PrCommentsPagePending structs and
  their *_comments_page_pending_matches staleness guards.
- Comment load/accept/failure now use PaginatedList::begin_page /
  accept_page / accept_failure with LoadCorrelation::Page.
- PaginatedList moved to src/domain/paginated_list.rs (src/state/pagination.rs
  is now a re-export shim) so domain detail types can own it without
  reversing the architecture DAG (domain must not depend on state).
- loading.comments retained as the screen-level spinner flag.
- GitHub parse layer seeds details with a settled PaginatedList.

No serialization migration: IssueDetail/PullRequestDetail are transient
(not persisted, not Serialize).

Behavior preserved incl. the BUG1 invariant: a failed comment page load
leaves next_page intact for retry (guarded by
failed_comment_page_preserves_continuation_for_retry). Stale comment pages
(wrong request_id or token) are rejected.

All 9 verification gates green: fmt, size, arch, clippy-strict,
clippy-complexity, coverage 72.57%, build, 2560 tests.

Closes #273.
